## 1. エグゼクティブサマリー

### 1.1 MVPの目的

COCOSiL MVPは、**「共感AI × 多角的性格分析で、ユーザーは継続的に利用するか」** という仮説を検証する。同時にBOOST第8期コンテストのデモとしても機能する。

### 1.2 検証仮説

| 優先度 | 仮説 | 成功指標 |
|--------|------|----------|
| ★ 最重要 | 性格分析を踏まえた共感AIチャットに、ユーザーは繰り返し戻ってくる | 7日以内の再訪率 30%以上 |
| ★ 重要 | MBTI×星座×動物占い×六星占術の統合レポートに「自分を理解された」と感じる | レポート満足度アンケート 4.0/5.0以上 |
| ★ 重要 | SNSシェアカードが拡散され、新規ユーザーを獲得する | シェア率 20%以上、シェア経由登録 5人以上 |
| 参考 | 共感チャット後のアクションプランを実行するユーザーがいる | アクションプラン「やってみた」報告率 10%以上 |

### 1.3 スコープ概要

| 項目 | 内容 |
|------|------|
| ターゲット | 25-35歳の若手社会人、職場の人間関係にストレスを抱える層 |
| プラットフォーム | Webアプリ（スマホブラウザ対応、モバイルファースト） |
| 技術スタック | Next.js / Supabase / Clerk / OpenAI Agent Builder / Gamma API |
| 開発体制 | **機能別分担**（後述）。AI Agent活用前提で作業領域が重複しない設計 |
| UI実装方針 | v0/ClaudeによるUI生成で効率化、shadcn/uiベース |
| 初期ユーザー | SNS小規模告知で50-100人想定 |
| 開発期間 | 4週間（4スプリント） |

## 3. 機能要件詳細

### 3.1 認証・オンボーディング

**機能仕様**

- Clerkによるソーシャルログイン（Google）またはメール/パスワード登録
- 初回ログイン時に生年月日入力画面を表示（星座・動物占い・六星占術の自動計算に必須）
- ニックネーム入力（任意、AIチャットでの呼びかけに使用）
- 生年月日確定時に、星座・動物占い(60アニマル)・六星占術タイプをサーバーサイドで即時計算しDBに保存

**技術メモ**

- ClerkのuserMetadataまたはSupabaseのprofilesテーブルに生年月日・各診断結果を格納
- 3つの診断計算はすべて生年月日のみで完結（ユーザー追加入力不要）

**担当: パートナー**

---

### 3.2 MBTI簡易診断

**機能仕様**

- 選択式クイズ: 8-12問、各問2選択肢
- 4軸（E/I, S/N, T/F, J/P）× 各軸2-3問でタイプ判定
- 「既に自分のMBTIタイプを知っている」ユーザー向けに **スキップボタン** → 16タイプから選択するオプションを提供
- 回答UI: スワイプまたはカード型で直感的に操作可能にする
- 選択済み状態の視覚的明確化（インタビューで指摘されたUI課題への対応）

**診断ロジック**

- 各軸のスコアを加算し、閾値で判定するシンプルな方式
- 各軸2問の場合、同点時はデフォルト値を設定する（例: E/I同点 → I判定）

**担当: えんまさ（質問設計・判定ロジック・UI、フロント〜バックエンド一気通貫）**

---

### 3.3 生年月日ベース診断（星座・動物占い・六星占術）

**機能仕様**

- **星座（12星座）:** 生年月日から自動判定。境界日処理を含むルックアップテーブル
- **動物占い（60アニマル）:** 生年月日から60分類のキャラクターを判定。12動物 × 各動物内の5分類で60パターン
- **六星占術（12タイプ）:** 生年月日から運命数を算出し、6タイプ（土星人・金星人・火星人・天王星人・木星人・水星人）× 霜(+)/霜(-)を判定

**全て計算ロジックは把握済み。** ユーザーの追加入力は一切不要。

**技術メモ**

- 星座: 日付範囲の定数テーブル（ライブラリ不要）
- 動物占い: ルックアップテーブルで60キャラ判定
- 六星占術: 運命数計算ユーティリティ関数
- テストケースで既知の結果（有名人等）と10件以上照合すること
- 全てオンボーディング時にサーバーサイドで計算し、profilesに保存

**担当: パートナー（3つの計算ロジック実装・テスト・DB保存、一気通貫）**

---

### 3.4 統合レポート（Lv3: 多面的解釈）— COCOSiLのコア技術

**これがCOCOSiLにしかない独自技術。** 4つの診断体系の結果を「矛盾も含めた1つのストーリー」に織り上げる。

**レポート構成**

1. **タイプサマリー:** 「あなたは INFJ × 蟹座 × 子守熊(コアラ) × 金星人(-)」
2. **統合プロフィール:** 4体系の共通点から導かれる「あなたの核心的特徴」（3-5行）
3. **多面的解釈:** 体系間の矛盾を「複雑さを誤解しない」解釈として提示
   - 例: 「MBTIのINFJ（内向的直感型）は1人の時間を大切にしますが、蟹座のあなたは人との深い繋がりも求めます。動物占いの子守熊タイプが示す慎重さと合わせると、"少数の本当に大切な人との、深い関係性"を最も重視するあなたの特徴が浮かび上がります」
4. **強みと注意点:** 各体系から示されるポテンシャルとリスク
5. **満足度アンケート:** レポート末尾に5段階評価「この分析はあなたに当てはまりましたか？」

**技術実装: LLM → Gamma APIパイプライン**

```
┌────────────────┐     ┌─────────────────┐     ┌──────────────┐
│  診断結果       │     │  LLM            │     │  Gamma API   │
│  (MBTI/星座/   │────▶│  アウトライン    │────▶│  リッチな     │
│   動物占い/    │     │  生成           │     │  レポート化   │
│   六星占術)    │     │  (構造化JSON)   │     │  (視覚的に    │
│                │     │                 │     │   美しい出力) │
└────────────────┘     └─────────────────┘     └──────────────┘
```

- **Step 1:** 診断結果をOpenAI APIに渡し、統合レポートのアウトライン（構造化JSON/Markdown）を生成
- **Step 2:** アウトラインをGamma APIに渡し、リッチなビジュアルレポートとして出力
- ナレッジベースに含めるデータ:
  - MBTI 16タイプの特性詳細（認知機能スタック含む）
  - 12星座の特性・属性（エレメント・クオリティ）
  - 動物占い60キャラの性格特性・行動パターン
  - 六星占術12タイプの特性・運気サイクル
  - 体系間の「よくある矛盾パターン」と解釈ガイドライン

**担当: えんまさ（プロンプト設計・ナレッジベース構築・Gamma API連携・UI、一気通貫）**

### 3.5 SNSシェアカード（独立機能）

**統合レポートとは別の独立機能として設計。** レポートの深いパーソナライズとは対照的に、シェアカードは「類型化」で拡散性を重視する。

**設計思想: レポートとシェアカードの役割分離**

| | 統合レポート | SNSシェアカード |
|--|-------------|----------------|
| 目的 | 深い自己理解（パーソナライズ） | SNS拡散・新規獲得（類型化） |
| 情報量 | 詳細（数百文字のストーリー） | 簡潔（キャッチーなタイプ名＋一言） |
| トーン | 内省的・深い | 楽しい・会話のきっかけ |
| 共有性 | 個人的（見せたくない人もいる） | オープン（見せたい） |

**機能仕様**

- 統合タイプ名を生成（例: 「直感の守護者」「情熱の調停者」など）
- キャッチーなビジュアルカード（タイプ名＋一言特徴＋4体系アイコン）
- **ワンタップシェア:** X（Twitter）/ Instagram Stories / LINEへの直接シェア
- シェアカードにはCOCOSiLのURL（招待リンク）を埋め込み
- レポート画面とは別に「あなたのカード」ボタンから生成・閲覧

**技術実装: LLM → Gamma APIパイプライン**

- **Step 1:** 診断結果からLLMで「統合タイプ名」と「一言キャッチコピー」を生成
- **Step 2:** Gamma APIでビジュアルカード画像を生成
- **Step 3:** OGP対応のシェアURLを動的生成（Next.js Dynamic OG Images or Gamma出力をそのまま利用）

**担当: パートナー（LLMプロンプト設計・Gamma API連携・シェアUI・OGP実装、一気通貫）**

- **システムプロンプト構成:**
  - ベースプロンプト: 共感AIとしての人格・トーン・3フェーズルール
  - ユーザーコンテキスト: 統合レポートの結果（4体系）を動的に注入

- **フェーズ遷移ロジック:** ターン数＋ユーザー発話の内容（悩み開示度合い）でフェーズ自動判定
- **ストリーミングレスポンス:** Server-Sent Events（SSE）でリアルタイム表示

**AIの人格設定（プロンプト設計の核心）**

- 温かみがあるが、媚びない。友人でも先生でもなく「信頼できる相談相手」
- 「占い」の世界観を活かしつつ、スピリチュアルに寄りすぎない
- ユーザーの性格タイプに応じてトーンを微調整（例: 思考型(T)にはロジカルに、感情型(F)には感情に寄り添って）

### 3.7 アクション記録

**機能仕様**

- Supabaseのaction_plansテーブルに保存

## 4. 役割分担: 機能別棲み分け

### 4.1 設計思想

**「機能別に完全分離し、作業領域を重複させない」**

お互いがAI Agentを活用して実装するため、細かい設計思想がリアルタイムで変化しやすい。レイヤー分担（フロント/バック）ではなく、**各人が担当機能のフロント〜バックエンド〜テストを一気通貫で持つ**ことで、Agent出力の衝突を回避する。

### 4.2 分担マップ

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  ┌─── えんまさ担当 ─────────────────────────────────────┐    │
```

### 4.3 分担の合理性

| 原則 | 実現方法 |
|------|---------|
| **作業領域の非重複** | 各機能は1人が完全オーナー。同じファイルを2人が同時に触らない |
| **AI Agent活用の最大化** | 各人が担当機能内で自由にAgent出力を反映できる。他人のコードとの整合を気にせず高速開発 |
| **接合点の最小化** | 機能間の連携はDBスキーマとAPI契約のみ。Sprint 1初日に合意し、以降は原則変更しない |
| **専門性の活用** | パートナー → インフラ・計算ロジック・シェア基盤。えんまさ → プロンプト設計・AI体験設計 |

### 4.4 事前合意事項（Sprint 1 Day 1に確定すること）

1. **DBスキーマ全テーブル定義** → 確定後は原則凍結。変更はPRでレビュー
2. **API契約書** → 各機能が公開するエンドポイントのリクエスト/レスポンス型定義
3. **Gitブランチ戦略** → `feature/F1-mbti`, `feature/F2-report` ... `feature/F7-infra` で分離
4. **共通UIコンポーネントの選定** → shadcn/uiのどのコンポーネントを使うか、テーマカラー等
5. **環境変数の命名規則** → `NEXT_PUBLIC_*`, `SUPABASE_*`, `OPENAI_*`, `GAMMA_*`

## 8. リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| 統合レポートのAI生成品質が低い | 高 | Sprint 2でプロンプトに最大工数を投下。品質不十分ならLv2にフォールバック |

| 共感チャットが「冷たい」と感じられる | 致命的 | Sprint 3でプロンプト反復テスト。インタビュー協力者に早期テスト依頼 |
| 機能別分担でのDB/API不整合 | 中 | Sprint 1 Day 1でスキーマ＋API契約を確定・凍結。変更はPRレビュー必須 |
| 4週間で終わらない | 中 | 最悪SNSシェアカードを落としても統合レポート＋共感チャットで成立 |
