# Antigravity 開発ガイドライン：ココシル V2 プロジェクト設計書

## 1. プロジェクトの背景と V2 の戦略的意義

本プロジェクトは、既存の「ココシル V1」を単にアップデートするものではなく、基盤から完全に再構築する「V2」としての新規リポジトリ開発である。

### 技術的な必然性と「リソース枯渇」への対策

V1 を更新せず V2 へ移行する最大の技術的要因は、既存環境における**「リソースの限界」**にある。V1 は現在、Supabase の無料枠である 0.5GB の容量制限に達し、書き込み不可のシステムダウン状態に陥っている。この「リソース枯渇」という致命的な失敗を教訓とし、V2 ではデータ設計の最適化と、シーケンス（処理順序）の根本的な変更を行う。ゼロベースでの再設計により、技術負債を清算し、開発の俊敏性を極限まで高めることが本プロジェクトの至上命題である。

### 戦略的転換点

V2 における転換点は、単なる機能追加ではなく「運用継続性」の確保にある。次セクションで定義する「API ファースト」の徹底により、フロントエンドの肥大化によるバックエンドの機能不全を未然に防ぐ。

---

## 2. API ファースト設計：型安全なコントラクトの義務化

開発効率を最大化し、実装の「手戻り地獄」を回避するため、本プロジェクトでは**「型安全なコントラクト（Type-Safe Contract）」**を基軸とした API ファースト設計を絶対的なプロトコルとする。

### 実装の複雑化リスクの排除

フロントエンドの UI デザインを先行させると、バックエンドの実装可能性を無視したボタンや機能が乱立し、ロジックが破綻する。これは V1 における開発遅延の主因であった。V2 では、API 仕様が確定し、型定義が共有されるまでフロントエンドの実装を開始することを禁止する。

### 設計フローと「コントラクト駆動開発」

1. **型定義（Type）の先行確定：** API のリクエスト/レスポンスの型を TypeScript で定義し、メインブランチにマージする。
2. **So What?（戦略的意義）：** これにより「コントラクト駆動開発」が可能になる。フロントエンド担当は、バックエンドの実装完了を待たずとも、合意された型に基づいた Mock データ（ダミー JSON 等）を使用して開発を進められる。この並行開発こそが、リソースの限られた本チームにおける加速装置となる。

---

## 3. ホールケーキの「横割り」：レイヤー別役割分担の定義

機能単位の「縦割り」分担は、依存関係のバッティングを招き、開発を停滞させる。本プロジェクトでは、技術レイヤーによる「横割り」の分担構造を採用し、各担当の専門性を最大化する。

### レイヤー別役割と責任範囲

* **バックエンド / インフラ（担当：平目）：**
  * **責務：** Supabase のセットアップ、DB スキーマ設計、Row Level Security (RLS) の設定、全 API の型定義、およびコアロジックの実装。
  * **So What?：** バックエンドがデータ構造とセキュリティ（RLS）を担保することで、システムの堅牢性を維持する。
* **フロントエンド / 要件定義（担当：遠藤）：**
  * **責務：** 詳細要件定義書の作成、UI/UX 実装、Server Components を活用した効率的なフロント実装。
  * **So What?：** 型定義という制約の中で、ユーザー体験（UX）のクオリティを最大化することに集中する。

この分担により、お互いのコードを破壊することなく、独立した開発サイクルを回すことが可能となる。

---

## 4. 技術スタックと標準開発プロトコル (SOP)

「ココシル V3」では、MVP 開発におけるスピードと検証精度を両立させるため、以下のモダンスタックと SOP を規定する。

### 標準技術スタック

* **Framework:** Next.js (App Router 必須)
* **Backend/DB:** Supabase
* **Deployment:** Vercel
* **Styling:** Tailwind CSS

### 環境構築 SOP（標準作業手順）

1. **開発環境の統一：** Windows 環境での開発は互換性リスクが高いため、Unix 系環境（Mac または WSL2）を必須とする。
2. **プロジェクト初期化：** `npx create-next-app` を実行。App Router, Tailwind CSS, TypeScript を必ず選択する。
3. **疎通確認：** `npm run dev` による localhost:3000 での起動確認後、直ちに GitHub リポジトリへ push する。
4. **環境変数の共有プロトコル：** Supabase の無料プランではチーム招待に制限があるため、URL および Anon Key は Discord の専用チャンネルを通じて安全に共有し、各自の `.env.local` に設定する。

## 5. 機能要件（F1〜F7）と実装ロードマップ

| ID | 機能名 | 担当 | 概要 | 依存関係 |
|---|---|---|---|---|
| F1 | MBTI 診断 | 遠藤 | ユーザー入力 UI。 | なし |
| F2 | レポート作成 | 遠藤 | LLM による分析。 | F5 の出力 |
| F3 | AI チャット | 遠藤 | 診断後の共感型対話。 | F2, F7 |
| F4 | 認証機能 | 平目 | ユーザー ID・セッション管理。 | なし |
| F5 | 生年月日診断 | 平目 | 星座・ロジック診断。 | なし |
| F6 | SNS シェアカード | 平目 | Gamma 等を用いた画像/スライド生成。 | F2 |
| F7 | 管理者ダッシュボード | 平目 | 履歴保存およびデータ管理。 | 全機能 |

### F7 (管理者ダッシュボード) の戦略的価値とデータ制約

F7 における「チャット履歴の保存」はサービス改善の核心であるが、V1 を崩壊させた 0.5GB 制限の主因でもある。

* **So What?：** F7 には、データの肥大化を防ぐための**「データ削除・アーカイブ戦略」**を初期設計から組み込む。非構造的なチャットログを無制限に保持せず、システムのアップタイムを最優先する。

## 6. データベース戦略とリソース制約の厳守

Supabase の無料枠という厳しい物理的制約下で運用を継続するため、以下のルールを徹底する。

### リソース管理プロトコル

* **プロジェクト数の制限：** 無料枠は 2 プロジェクトまで。V2 用のリソースを確保するため、**既存の不要なプロジェクト（「インサイダーゲーム」等）は直ちに停止（Pause）**すること。
* **容量の死守：** チャット履歴等のテキストデータは、インデックス設計を最小限に留め、0.5GB を超過させない。
* **セキュリティ共有：** チーム招待機能を使わず「Anon Key」を共有するため、環境変数の管理には最大限の注意を払うこと。リポジトリへの環境変数の混入は厳禁とする。

## 7. 次のステップ：MVP「簡易診断」へのフォーカス

プロジェクトを軌道に乗せるための最初のマイルストーンを「MVP 簡易診断」のエンドツーエンド（E2E）疎通に設定する。

### 直近のアクションアイテム

* **平目（Hirame）：**
  * Supabase データベースの新規作成と接続情報の共有。
  * 全 API（F1〜F7）の Request/Response 型定義の作成。
  * 「簡易診断 API」のプロトタイプ実装。
* **遠藤（Endo）：**
  * F1〜F7 の各機能に関する「詳細要件定義書」の作成。
  * 平目の定義した型に基づいた、診断 UI のプロトタイプ実装。

この最初の開発サイクルを最速で回し、Antigravity の開発リズムを確立せよ。本ガイドラインの不履行は、V1 の失敗を繰り返す技術的怠慢とみなす。
