# システム要件定義書

## システム概要

### 1. システムの目的

COCOSiL（ココシル）は、職場の人間関係にストレスを抱える25-35歳の若手社会人を主なターゲットとした、AI共感チャット×多角的性格分析プラットフォームである。「自分を知って、ラクになる」をコンセプトに、MBTI・星座・動物占い（60アニマル）・六星占術の4つの診断体系をAIが統合分析し、矛盾も含めた多面的な自己理解を提供する。さらに、診断結果を踏まえた「根拠のある共感」を持つAIチャットにより、従来の占いアプリやカウンセリングでは満たされなかった「気軽に、いつでも、本質的に寄り添ってくれる存在」への需要に応える。

- **解決する具体的な課題:** 職場の人間関係ストレスを抱えながら、高額カウンセリング（月数万円）にはアクセスしづらく、既存の占いアプリには実用性がなく、汎用AIチャット（ChatGPT等）では共感が浅いという3重のギャップ
- **目標とする成果:** MVP公開後4週間で、7日以内再訪率30%以上を達成し、「共感AI×性格分析」の体験がユーザーの継続利用を生むことを検証する
- **期待されるビジネスインパクト:** BOOST第8期コンテストでの評価獲得、SNS経由でのバイラル獲得の実証、フリーミアム課金モデル（500-1,500円/月）の検証基盤構築

### 2. 主な機能

**コア機能（4個）:**
- F1: MBTI簡易診断（8-12問の選択式クイズ＋自己申告スキップ）
- F2: 4体系統合レポート生成（MBTI×星座×動物占い×六星占術のLv3多面的解釈、LLM→Gamma APIパイプライン）
- F3: 共感AIチャット（3フェーズ設計: 共感→性格分析を交えた深掘り→アクションプラン提示）
- F5: 生年月日ベース自動診断（星座/動物占い60アニマル/六星占術を生年月日のみで即時計算）

**差別化機能（2個）:**
- F2の統合エンジン: 4体系間の矛盾を「複雑さの肯定的解釈」として統合するAIプロンプト技術（世界初レベル）
- F6: SNSシェアカード（統合タイプ名＋ビジュアルカードをワンタップでX/Instagram/LINEにシェア、招待リンク埋め込み）

**付加価値機能（2個）:**
- F3拡張: アクションプラン記録（「やってみた」ボタンによる再訪ループ）
- F7: 簡易アナリティクス＋管理者ダッシュボード

### 3. 提供する価値

- **ユーザーにとっての直接的価値:** 24時間いつでも、性格を理解した上で寄り添ってくれるAI相談相手。深夜の感情発散（りこペルソナ）から月1回の占い師セッション間の29日間を埋める存在（高橋さんペルソナ）まで対応
- **ビジネスにとっての価値:** 4体系統合×AI共感という独自技術による参入障壁の構築、SNSシェアによるバイラル獲得チャネル、フリーミアムモデルへの課金転換基盤
- **社会的価値:** 高額カウンセリングにアクセスできない層への低コストなメンタルヘルス支援手段の提供（ただし医療行為の代替ではないことを明示）

### 4. 実現方法の概要

- **主要なテクノロジー要素:** Next.js（App Router）/ Supabase（PostgreSQL + RLS）/ Clerk（認証）/ OpenAI Agent Builder（チャット・分析）/ Gamma API（レポート・カード生成）
- **重要なインテグレーションポイント:** LLM→Gamma APIパイプライン（統合レポート・SNSカード生成）、OpenAI SSE→フロントエンドリアルタイム表示、Clerk→Supabase認証連携
- **スケーラビリティへの考慮:** MVP段階では50-100人規模。Supabase無料枠（0.5GB制限）の制約を認識しつつ、Vercelサーバーレス構成により水平スケールの基盤は確保

## 機能要件

### FR-01: 認証・オンボーディング（F4）

| 項目 | 内容 |
|------|------|
| 概要 | Clerkによるソーシャルログイン/メール登録＋初回生年月日・ニックネーム入力 |
| 入力 | Google OAuth or メール/パスワード、生年月日（必須）、ニックネーム（任意） |
| 出力 | 認証済みユーザーセッション、profilesレコード作成 |
| 処理 | 認証後、生年月日から星座・動物占い・六星占術を即時自動計算しDB保存 |
| 受入基準 | ログイン完了からプロフィール画面表示まで3秒以内。生年月日未入力では診断画面に進めない |

### FR-02: MBTI簡易診断（F1）

| 項目 | 内容 |
|------|------|
| 概要 | 8-12問の選択式クイズによる16タイプ判定。既知ユーザー向けスキップ（自己申告）対応 |
| 入力 | 各問2択の回答 or 16タイプからの直接選択 |
| 出力 | MBTIタイプ（4文字）、各軸スコア |
| 処理 | 4軸（E/I, S/N, T/F, J/P）×各2-3問のスコア加算＋閾値判定。同点時デフォルト値 |
| 受入基準 | 診断完了まで5分以内。選択済み状態が視覚的に明確（インタビュー指摘対応） |

### FR-03: 生年月日ベース診断（F5）

| 項目 | 内容 |
|------|------|
| 概要 | 生年月日のみから星座（12）、動物占い（60アニマル）、六星占術（12タイプ）を自動計算 |
| 入力 | 生年月日（DATE型） |
| 出力 | zodiac_sign, animal_type, animal_group, rokusei_type |
| 処理 | 星座=日付範囲テーブル、動物占い=ルックアップテーブル、六星占術=運命数算出関数。全てサーバーサイド |
| 受入基準 | 有名人の生年月日10件以上で既知の結果と照合しパス。計算は500ms以内 |

### FR-04: 統合レポート生成（F2）— コア技術

| 項目 | 内容 |
|------|------|
| 概要 | 4体系の診断結果をLLMで統合分析し、Gamma APIでリッチなビジュアルレポートとして出力 |
| 入力 | MBTI/星座/動物占い/六星占術の各タイプ＋ナレッジベース |
| 出力 | 構造化アウトライン（JSON）、Gammaレポート閲覧URL |
| 処理 | Step1: OpenAI APIで統合アウトライン生成（Lv3多面的解釈）→ Step2: Gamma APIでレポート化 |
| レポート構成 | (1)タイプサマリー (2)統合プロフィール (3)多面的解釈（矛盾含む） (4)強みと注意点 (5)満足度アンケート |
| 受入基準 | 5パターン以上のタイプ組み合わせでテストし、満足度4.0/5.0以上。レポート生成15秒以内 |

### FR-05: SNSシェアカード（F6）

| 項目 | 内容 |
|------|------|
| 概要 | 統合タイプ名＋キャッチコピーのビジュアルカード生成、ワンタップSNSシェア |
| 入力 | 4体系の診断結果 |
| 出力 | type_name, catchphrase, Gammaカード画像URL, シェアURL（招待リンク埋め込み） |
| 処理 | Step1: LLMでタイプ名＋キャッチコピー生成 → Step2: Gamma APIでカード生成 → Step3: OGP動的生成 |
| 受入基準 | X/Instagram/LINEの各プラットフォームでシェア確認。OGP画像が正しく表示される |

### FR-06: 共感AIチャット（F3）— MVPの心臓部

| 項目 | 内容 |
|------|------|
| 概要 | 4体系の性格分析結果を踏まえた3フェーズ共感チャット |
| 入力 | ユーザーメッセージ、統合レポート結果（コンテキスト注入）、会話履歴 |
| 出力 | AIレスポンス（SSEストリーミング）、フェーズ状態、抽出されたアクションプラン |
| 3フェーズ | Phase1: 共感・傾聴（3-5ターン）→ Phase2: 性格分析を交えた深掘り（3-5ターン）→ Phase3: アクションプラン提示（1-2ターン） |
| 設計原則 | 「共感→安心→分析」の順番を絶対に守る。いきなり質問攻め・解決策提示はNG |
| 受入基準 | Phase遷移が自然に行われる。応答開始まで2秒以内（SSEストリーミング）。平均8ターン以上の会話継続 |

### FR-07: アクション記録

| 項目 | 内容 |
|------|------|
| 概要 | チャットで提示されたアクションプランの一覧表示＋実行記録 |
| 入力 | AIレスポンスから構造化抽出（JSON mode）されたアクション |
| 出力 | アクション一覧、完了状態、タイムスタンプ |
| 処理 | 次回チャット開始時にAIが「前回のアクション」に自然に触れる |
| 受入基準 | 「やってみた」ボタン押下で即座にDB更新。チャットでの言及が自然 |

### FR-08: 簡易アナリティクス（F7）

| 項目 | 内容 |
|------|------|
| 概要 | 管理者向けKPIダッシュボード |
| 計測指標 | DAU、7日以内再訪率（最重要）、チャットセッション数、診断完了率、アクション実行率、レポート満足度、SNSシェア率、シェア経由登録数 |
| 受入基準 | 主要KPIがリアルタイム（1時間以内の遅延許容）で管理画面に表示される |

### NFR-02: セキュリティ

| 項目 | 要件 | 根拠 |
|------|------|------|
| 認証 | Clerk（OAuth 2.0 / JWT）による認証。全API RoutesでJWT検証必須 | 個人情報（生年月日・悩み相談内容）を扱うため |
| データアクセス制御 | Supabase RLS（Row Level Security）により全テーブルでユーザーは自分のデータのみアクセス可能 | チャット内容は極めてセンシティブ |
| 通信暗号化 | HTTPS必須（Vercelデフォルト） | 標準要件 |
| APIキー管理 | OpenAI/Gamma/Supabaseキーは環境変数で管理。クライアントに露出させない | APIキー漏洩防止 |
| シェアカードの公開範囲 | share_cardsテーブルのみRLSで公開読み取り許可（シェアURL経由） | SNSシェア機能に必要 |
| 医療免責 | アプリ内に「本サービスは医療行為の代替ではありません」の免責事項を常時表示 | 法的リスク回避 |

### NFR-03: 可用性・信頼性

| 項目 | 目標値 | 根拠 |
|------|--------|------|
| サービス稼働率 | 99%（月間ダウンタイム7.2時間以内） | MVP段階。Vercel+Supabaseのマネージドサービス依存 |
| データバックアップ | Supabaseの日次自動バックアップ（無料枠で利用可能） | チャット履歴の消失はUX致命的 |
| エラー通知 | Vercel Logs＋メール通知（Supabase Webhook等） | 障害早期検知 |

### NFR-04: スケーラビリティ

| 項目 | 設計方針 | 根拠 |
|------|---------|------|

| DB設計 | テーブル設計時にインデックス戦略を考慮。user_id外部キーに自動インデックス | クエリパフォーマンス維持 |

### NFR-05: 保守性

| 項目 | 設計方針 | 根拠 |
|------|---------|------|

| 環境変数管理 | `NEXT_PUBLIC_*`, `SUPABASE_*`, `OPENAI_*`, `GAMMA_*` の命名規則統一 | キックオフ合意事項 |
| バージョン管理 | GitHub。機能別ブランチ（feature/F1-mbti等） | 2人開発での衝突回避 |

## 技術スタック

| レイヤー | 技術 | 選定理由 |
|---------|------|---------|
| **フロントエンド** | Next.js 14+（App Router） | SSR/SSG/APIルート統合、Vercelとのネイティブ連携、v0によるUI生成との親和性。チーム（遠藤）の経験あり |
| **UIフレームワーク** | shadcn/ui + Tailwind CSS | v0/Claude生成UIとの互換性が高い。コンポーネントベースでカスタマイズ容易。バックエンド寄りチームでもUIを構築しやすい |
| **認証** | Clerk | Google OAuth＋メール認証をSaaS提供。Supabase RLSとの連携実績あり。MAU 10,000まで無料 |
| **データベース** | Supabase（PostgreSQL） | リアルタイム機能、RLS、REST API自動生成。チーム（ヒラメ）のV2での使用経験あり。無料枠でMVP運用可能 |
| **AI/LLM** | OpenAI Agent Builder（GPT-4o相当） | 共感チャットの品質、構造化出力（JSON mode）対応、ストリーミング対応。Agent Builderによるプロンプト管理 |
| **レポート生成** | Gamma API | LLMアウトラインからリッチなビジュアルレポート/カードを生成。APIキー取得済み |
| **ホスティング** | Vercel | Next.jsのネイティブホスティング。サーバーレス関数、エッジ配信、自動HTTPS。無料枠でMVP運用可能 |
| **ストリーミング** | Server-Sent Events（SSE） | チャットのリアルタイム応答。WebSocketより実装がシンプルで、Next.js API Routesとの相性が良い |

### 技術選定における制約事項

- Supabase無料枠: 0.5GBストレージ、2プロジェクト制限（キックオフで確認済み。ヒラメが既存プロジェクトを一時停止して対応）
- Gamma API: レート制限・コスト・レスポンス速度が公式ドキュメントで未確認（Sprint 2早期にPoC実施が必須）
- OpenAI APIコスト: 月額1万円以内の運用を想定。チャット回数制限（無料枠月5セッション等）で制御

**バックエンドファースト**（キックオフ決定）: APIの入出力の型を先に定義し、安定した開発基盤を構築する。フロントエンド主導で始めるとバックエンド実装が複雑化するという過去の経験に基づく。

## リスク分析

### 技術的リスク

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|---------|------|
| 統合レポートのAI生成品質が低い | 高 | 中 | Sprint 2でプロンプトに最大工数投下。5パターン以上テスト。品質不十分ならLv2にフォールバック |
| Gamma APIの出力品質/速度が期待に合わない | 高 | 中 | Sprint 2早期にPoC実施。不適合なら静的HTML/CSSレポートにフォールバック |
| 共感チャットが「冷たい」「機械的」と感じられる | 致命的 | 中 | Sprint 3でプロンプト反復テスト。インタビュー協力者に早期テスト依頼 |
| Supabase無料枠（0.5GB）超過 | 中 | 低 | チャットメッセージの圧縮、古いセッションのアーカイブ。超過時はProプランに移行（月$25） |
| SSEストリーミングのVercelタイムアウト | 中 | 中 | Vercel関数のmaxDuration設定。長時間会話は分割レスポンス |

### ビジネス的リスク

| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|---------|------|
| 初期ユーザー50人を集められない | 高 | 中 | SNS告知に加え、インタビュー協力者への直接依頼。BOOSTコミュニティ内での試用依頼 |
| 再訪率30%未達（仮説不成立） | 高 | 中 | チャット品質の問題かコンセプトの問題かを切り分けるため、アンケート＋定性インタビューを実施 |
