# [F1] MBTI簡易診断 詳細要件定義書

## 1. プロジェクト概要
COCOSiL（ココシル）は、職場の人間関係にストレスを抱える25-35歳の若手社会人をターゲットとした「自分を知って、ラクになる」をコンセプトとするAI共感チャット×多角的性格分析プラットフォームである。
本ドキュメントでは、コア機能の一つである「[F1] MBTI簡易診断」の詳細要件を定義する。この機能は、ユーザーが自身の性格タイプを把握し、続く統合レポート生成（F2）および共感AIチャット（F3）のための重要かつ基礎的なコンテキスト情報を収集することを目的とする。

## 2. ビジネス要件
- **目的**: ユーザーに素早く自身の性格タイプを認識させ、サービスに対する没入感と納得感を高める。
- **KGI/KPI**:
  - 診断完了率: **70%以上**（オンボーディング完了後、離脱せずに診断を終える割合）
  - ユーザーの初期エンゲージメント獲得と、続く機能へのスムーズな誘導。
- **制約事項**: ユーザーの入力負担を最小限に抑えるため、質問数を絞り込みつつ、既存の診断結果を持っているユーザーには即時回答（スキップ）を許容する。

## 3. ユーザー要件
- **ターゲットユーザー**: 職場の人間関係にストレスを抱える25-35歳の若手社会人（例：りこ、高橋さん など）。
- **ユーザーの抱える課題**: 
  - 長時間の診断テストは面倒で途中で離脱してしまう。
  - 既に自身のMBTIを知っている場合、再度数十問のテストを答えるのは苦痛。
- **ユーザーストーリー**:
  1. 初回ログインユーザーは、楽しく直感的なUI（スワイプやカード型）でサクサクと8-12問の質問に答え、自分のタイプを知ることができる。
  2. 既に自分のMBTIを知っているユーザーは、「スキップ」ボタンから直接16タイプを選択し、すぐに統合レポート生成へと進むことができる。

## 4. 機能要件

### 機能一覧
| 機能ID | 機能名 | 概要 |
|--------|--------|------|
| F1-1 | 診断画面表示 | 8-12問の選択式クイズ（各2択）を表示するUI（カード型/スワイプ等） |
| F1-2 | 回答入力・状態表示 | 選択済み状態の視覚的な明確化、次の質問への遷移 |
| F1-3 | スキップ・直接選択 | 「既にMBTIを知っている」ユーザーが16タイプから直接選択する機能 |
| F1-4 | 判定ロジック処理 | E/I, S/N, T/F, J/Pの4軸に基づくスコア加算と閾値判定処理（同点時はデフォルト値を設定） |
| F1-5 | 結果保存 | 判定されたMBTIタイプ（4文字）および各軸スコアをDBへ保存 |

### 機能詳細仕様・ユースケース

#### ユースケース1: 新規診断フロー（通常フロー）
- **アクター**: 新規ユーザー
- **事前条件**: ユーザーがオンボーディング（Clerk認証＋生年月日入力）を完了していること。
- **正常系フロー**:
  1. システムは診断開始画面を表示し、第1問を出題する。
  2. ユーザーは2択から回答を選択する。
  3. システムは選択状態を視覚的にフィードバックし、直ちに次の問題を提示する。
  4. 8〜12問すべての回答が完了する。
  5. システム（API）は回答結果から4軸のスコアを加算し、閾値判定にて16タイプを決定する。
  6. システムは `diagnoses` テーブルおよび `profiles` テーブル（`mbti_type`）に結果を保存する。
  7. システムは統合レポート作成画面（F2）へ自動遷移させる。
- **例外系フロー**:
  - 回答途中で通信エラーが発生した場合、エラーメッセージを表示し、再試行または途中からの再開を促す。

#### ユースケース2: 診断スキップフロー
- **アクター**: 既に自身のMBTIタイプを知っているユーザー
- **正常系フロー**:
  1. システムは診断開始画面に「既に自分のMBTIタイプを知っている」スキップボタンを提示する。
  2. ユーザーがスキップボタンを押下する。
  3. システムは16タイプの一覧（ドロップダウン等）を表示する。
  4. ユーザーが自身のタイプを選択し、確定する。
  5. システムは `diagnoses` テーブル等に直接入力結果として保存し、レポート作成画面（F2）へ遷移させる。

## 5. 非機能要件
- **パフォーマンス**:
  - 画面遷移・次問表示時のレスポンスタイム：**500ms以内**（スワイプ/カード型UXの快適さ維持）。
  - 診断完了（全問終了〜結果画面遷移）：**5分以内**（ユーザーの負担軽減）。
  - APIレスポンスタイム（判定保存処理）：**2秒以内**。
- **可用性**: 常に安定して診断が完了できるようVercelやSupabaseのアップタイム（99%）に依存する。
- **保守性**: 判定ロジックの質問追加・変更が容易に行えるよう、質問データはロジックから分離し構造化して管理する。

## 6. インテグレーション要件
- **連携API**: 
  - エンドポイント: `POST /api/diagnosis/mbti`
  - 役割: MBTI回答を受け取り、判定ロジックの実行、結果（MBTIタイプ、各軸スコア等）を算出。
- **データ連携 (DB)**:
  - 判定結果をSupabaseに保存。
  - `diagnoses` テーブルに `mbti_answers`（回答内容）, `mbti_scores`（各軸のスコア）を保存。
  - `profiles` テーブルに `mbti_type`（4文字の文字列、例:'INFJ'）を保存。

## 7. 技術選定とアーキテクチャ
- **フロントエンド UI**: Next.js App Router, React コンポーネント, shadcn/ui, Tailwind CSS を用いて直感的かつリッチなUI（カード/スワイプ型など）を構築する。
- **バックエンド API**: Next.js API Routes 上のサーバーサイドにてスコア計算ロジックを実行。
- **データベース**: Supabase (PostgreSQL) を使用し、Prisma 等のORM/型安全クライアントまたは直接Supabase Clientを使用してデータをセキュア（RLS適用）に保存する。

## 8. 開発プロセス / スケジュール
- **担当**: えんまさ（質問設計・判定ロジック設計・UI実装・API実装まで一気通貫で担当）
- **マイルストーン (Sprint 1 内)**:
  - Day 1迄: DBスキーマ・APIのエンドポイント型の定義（事前合意に準ずる）
  - 中盤: MBTI設問の設計、MBTI UIの実装・判定ロジックの実装
  - 完了条件: 全機能の疎通確認。ユーザーが診断ロジックを通してMBTI結果をDBに正しく保存できること。

## 9. リスクと課題
- **診断精度のトレードオフ**: 8-12問という少ない質問数では正確なMBTI診断が難しくなる可能性がある。(仮定) 今回は「簡易診断」かつ共感チャットへの導入としての性格付けが目的であるため、学術的正確さよりもユーザー体験・手軽さを優先する。
- **UIの実装 난易度**: スワイプやカード型のインタラクション実装が遅延するリスクがある。必要に応じてshadcn/uiの基本コンポーネントを用いたシンプルなラジオボタン＋次へボタンのUIへフォールバックする。

## 10. ランニング費用と運用方針
- 質問データと判定ロジックはアプリケーションのコードベース内または定数データとして管理し、外部APIへのリクエストは発生しない。
- Supabaseの無料枠内で運用するため、`diagnoses` のレコードサイズなどは最適化を保つ。

## 11. 変更管理
- V1.1方針に基づき、バックエンドとフロントエンドを分離せず、「[F1]機能オーナー」としてえんまさがエンドツーエンドで実装・管理を行う。
- テーブル定義やAPIコントラクトに変更が必要な場合は、Sprint 1の初段にてPRレビューを通じ他メンバー（ヒラメ）と合意形成を行う。
